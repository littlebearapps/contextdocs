# Documentation Standards

When generating public-facing repository documentation, follow these principles.

## The 4-Question Test

Every document must answer: (1) Does this solve my problem? (2) Can I use it? (3) Who made it? (4) Where do I learn more?

## The Lobby Principle

The README is the lobby — enough to decide whether to enter, not the whole building. Value proposition, quick start, top features, comparison, badges, and links belong in README. Detailed guides, tutorials, config references, and architecture belong in `docs/guides/`. If a section exceeds 2 paragraphs or an 8-row table, delegate it.

## Time to Hello World

State TTHW explicitly where evidence supports it. Concrete before abstract, one concept per step, all commands copy-paste-ready.

## Tone & Language

Follow the project's locale. Benefit-driven, reader-centric ("You can now..." not "We implemented..."). Active voice, short sentences.

**Banned phrases:** "in today's digital landscape", "it's important to note", "dive into", "leverage", "game-changer", "cutting-edge", "seamless", "robust", "in conclusion", "furthermore", "revolutionise", "utilise", "comprehensive", "navigate the complexities", "elevate your". No "simple"/"easy"/"powerful" without evidence.

## Feature-to-Benefit Writing

Pattern: `[Feature] so you can [outcome] — [evidence]`. Every feature needs evidence. Use at least 3 of: time saved, confidence gained, pain avoided, capability unlocked, cost reduced. Load `feature-benefits` skill for the full framework.
